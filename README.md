## 2️⃣ 프로젝트 주제 및 선정 이유

### 2‑1️⃣ 주제

> _예시) “은행 고객 이탈(Churn) 예측”_

### 2‑2️⃣ 주제를 선택한 이유

- **비즈니스 가치** – 이탈 고객을 사전에 파악해 맞춤형 마케팅/유지 전략을 수립할 수 있음.
- **데이터 가용성** – Kaggle·공공 데이터에 풍부한 고객 거래·행동 로그가 존재.
- **기술 학습 목표** – LightGBM·앙상블·SHAP 해석 등 최신 ML 기법을 실전 프로젝트에 적용하고 싶음.

---

## 3️⃣ 주요 기능

- **이탈 확률 예측** – 입력된 고객 정보(거래·인구통계)로 이탈 가능성을 실시간 예측.
- **시각화 대시보드 (Streamlit)** – 전체 고객 이탈 분포, 개별 고객 상세 리포트, 모델 성능 지표(AUC, F1 등).
- **예측 기반 알림** – 이탈 위험 고객에게 자동 메시지(예: 이메일/카카오톡) 전송 (선택 구현).

## 7️⃣ Application 주요 기능

### 7‑1️⃣ 제공 기능

- **입력 폼** – 고객 기본·거래 정보를 입력받는 UI.
- **예측 버튼** – 클릭 시 LightGBM 모델이 이탈 확률을 반환.
- **리포트 페이지** – 예측 결과와 주요 피처 SHAP 해석을 시각적으로 표시.

### 7‑2️⃣ 기술 스택

- **백엔드**: Python 3.11, LightGBM, joblib (모델 직렬화)
- **프론트엔드**: Streamlit, Plotly, CSS (다크모드·그라데이션)

## 8️⃣ 회고

### 8‑1️⃣ 구현 중 발생한 문제 & 해결 방안

| 문제                       | 원인                                        | 해결 방법                                                   |
| -------------------------- | ------------------------------------------- | ----------------------------------------------------------- |
| 결측치·이상치 처리         | 일부 컬럼(`Education_Level` 등) 데이터 누락 | 평균/최빈값 대체 + 이상치 IQR 필터링                        |
| 클래스 불균형 (양성 16%)   | 데이터 자체 비율                            | `scale_pos_weight` 조정 + Stratified K‑Fold CV              |
| 모델 저장/로드 오류        | `joblib` 버전 차이                          | `joblib==1.4.0` 고정, `pickle_protocol=4` 사용              |
| Streamlit UI 레이아웃 깨짐 | CSS 충돌                                    | `st.markdown`에 `unsafe_allow_html=True` 로 커스텀 CSS 적용 |
| …                          | …                                           | …                                                           |

### 8‑2️⃣ 팀원별 느낀 점

- **김준석**: “데이터 전처리 파이프라인을 자동화하면서 파이썬 pandas 활용 능력이 크게 향상되었습니다.”
- **문지영**: “이번 이탈률 예측 프로젝트를 진행하면서, 단순히 모델을 학습시키는 것보다 데이터를 어떻게 다루고 설계하느냐가 모델 성능에 더 큰 영향을 준다는 것을 직접 체감할 수 있었다. 처음 모델을 학습시켰을 때는 클래스 불균형 문제로 인해 모델이 특정 클래스에 치우쳐 예측하는 현상이 나타났다. 이를 해결하기 위해 SMOTE를 활용해 Minority Class를 증강했고, 그 결과 모델이 두 클래스를 보다 균형 있게 학습할 수 있도록 개선되었다. 또한, 단순히 원본 컬럼만 사용했을 때보다, 도메인 기반의 파생 변수를 추가한 후 모델의 표현력이 확실히 개선되는 경험을 했다. 단순히 데이터를 사용하는 것이 아니라, 의미 있는 정보를 재구성하거나 만들 수 있다는 점이 성능 향상에 큰 역할을 한다는 것을 배웠다. 최적화된 단일 모델도 한계를 가질 수 있으며, 서로 다른 오류 패턴을 가진 모델(LightGBM, Random Forest, Logistic Regression 등)을 조화롭게 결합했을 때 가장 안정적이고 높은 성능을 얻을 수 있음을 배웠다. 앙상블은 단순히 성능을 높이는 것을 넘어, 모델의 예측 결과가 극단적으로 흔들리는 것을 방지하는 안정성을 제공함을 체감했다. 이번 프로젝트를 하면서 기술적인 성과뿐 아니라, 협업의 의미도 다시 한 번 느꼈다. 아직 부족한 부분이 많았지만, 팀원들이 방향을 잡아주고 함께 고민하며 진행한 덕분에 다양한 시도와 배움을 얻을 수 있었다. ”
- **신병탁**: “LightGBM 하이퍼파라미터 튜닝 과정이 가장 도전적이었지만, 성능 향상을 직접 체감했습니다.”
- **이명준**: “프로젝트 전체 흐름을 관리하면서 Git·CI/CD 워크플로우를 정리하는 것이 중요함을 깨달았습니다.”
- **손현우**: "Feature Engineering을 통하여, 코드를 하나하나 확실하고 섬세하게 테스트하며 만들어야되는 중요성을 배웠습니다."
